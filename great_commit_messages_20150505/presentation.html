Great Commit Messages

Today, I'd like to talk about Great Commit Messages: + Why they're important + The 7 Rules of Writing Them

Why

When it comes to writing great commit messages, "Why" is both the question and the answer. Let's start with the question:

Why version control?

Why do we use version control? (What if we didn't?) ... + System of Record + Collaboration + Coordination + Authoritative Source on the Code

Version control is important

I don't think anybody here will disagree with this statement

Show of hands:
 People who have looked at project history

Who here has ever looked at a project's history? Maybe you were troubleshooting a bug; Maybe you were looking for an explanation (Nearly Unanimous!)

Why?

Why (for what purposes) have you looked at source control history? + Compare changes to avoid conflicts + Figure out what changed to troubleshoot bugs + Look at a previous commit to find out what went into a feature + Who wrote this line of code so I can ask them a question? + Code Quality: every time you do a pull request, you look at history!

Repository of Knowledge

Source control history is Repository of Knowledge, not just of code.

Code tells you what changed and how
Great commit messages tell you what changed why

The secret to writing great commit messages is to share knowledge, and to explain WHY. Also What. If you do this, you will impart the following benefits to future you and your whole team:

Switch contexts faster
Make well-informed decisions
Make them quicker

If your commit messages explain why, + You can switch contexts faster + Your decisions will be well-informed + You can make decisions quicker In other words:

Less time seeking answers
More time creating answers

Basically, you spend less time seeking answers and more time creating them. In order to do that, we need a standard for conveying information in the git history

7 Rules of Great Commit Messages

I did not come up with these, but I like them. So does everybody else on the whole internet. BUT FIRST, an example:

The first FIVE deal with the subject line.

1. Separate subject from body with a blank line
2. Limit the subject line to 50 characters
3. Capitalize the subject line
4. Do not end the subject line with a period
5. Use the imperative mood in the subject line
6. Wrap the body at 72 characters
7. Use the body to explain what and why vs. how
How to make great commit messages?
Think about your Audience
Your teammates
Future developers
Future you
Empower them to make the best decisions
Suggestions:
Tag issues / Add URLs in the description

Give context to the changes you're making. If you're working on a story logged in Rally or Github, specifying the issue number will help future developers understand the story behind your changes.

Explain Motivation

This can sometimes be self-explanatory, but it never hurts to provide context "Remove trailing commas to fix JSHint" as opposed to just "Remove trailing commas" The benefit with this is that we can begin to index and analyze the time/commits invested in certain subjects

Explain Tradeoffs

Software development is a constant exercise in picking from a set of tradeoffs. When modifying a piece of code, knowing the tradeoffs at play can save time and effort, resulting in higher quality software

Explain Other Approaches

This gives future developers more context to make informed decisions about changes to the code This is especially helpful when tradeoffs are being made "in the interest of time"

Discuss Difficulties

This information may help future developers who run into similar problems, and may also facilitate discussions about tradeoffs

Mention relevant commits

When fixing bugs or removing old code, Sticking the hash in the commit message can help close the loop If issues with YOUR code come up EX: Trying to increase max zoom level in stacks
